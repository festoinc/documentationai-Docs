## Overview

The BookingPal Supplier API uses **two-layer authentication** for all requests:

1. **JWT Token** — passed as a `jwt` query parameter, obtained via the Login endpoint.
2. **API Key** — passed as an `x-api-key` HTTP header, provided by BookingPal during onboarding.

<Callout kind="alert">
All API calls (except incoming push/webhook endpoints) require **both** a valid JWT token **and** an `x-api-key` header. Tokens expire after **120 minutes**.
</Callout>

## How Authentication Works

<Steps>
  <Step title="Login" icon="key">
    Call `GET /authc/login` with your `username` (email) and `password` as query parameters. The response contains a JWT token.
  </Step>
  <Step title="Use both credentials" icon="send">
    Include the JWT token as a `jwt` query parameter **and** your API key as an `x-api-key` header in every subsequent API request.
  </Step>
  <Step title="Refresh when expired" icon="refresh-cw">
    Tokens are valid for 120 minutes. Call the Login endpoint again to obtain a new token before the current one expires. The `x-api-key` header remains the same.
  </Step>
</Steps>

## Login Endpoint

| Field | Value |
|---|---|
| **Method** | `GET` |
| **URL** | `/authc/login` |
| **Auth** | None (credentials in query params) |

<ParamField query="username" param-type="string" required="true">
  Your account email address (PMS or PM).
</ParamField>

<ParamField query="password" param-type="string" required="true">
  Your account password.
</ParamField>

**Example Request:**

```bash
curl -G "https://apidemo.mybookingpal.com/authc/login" \
  --data-urlencode "username=your-email@example.com" \
  --data-urlencode "password=your-password"
```

**Example Response:**

```json
{
  "token": "d5d308244d9eb48a/e9073f6d-85a1-49b8-a799-19c57b66fb24",
  "partyId": 61692799,
  "organizationId": 61690135,
  "name": "Riverwalk Inn",
  "currency": "USD",
  "supplierId": 61692799,
  "is_error": false
}
```

## Using the Credentials

Every authenticated request requires **both** credentials:

- `jwt` query parameter — your session token
- `x-api-key` header — your permanent API key

```bash
curl -G "https://apidemo.mybookingpal.com/pmlist" \
  -H "x-api-key: YOUR_API_KEY" \
  --data-urlencode "jwt=YOUR_TOKEN_HERE"
```

<Callout kind="tip">
Implement token caching in your application. Store the token and its expiration time. Re-authenticate only when the token is about to expire. The `x-api-key` header stays constant across sessions.
</Callout>

## Token Types

<Tabs>
  <Tab title="PMS-Level Token" icon="building">
    Obtained using **PMS credentials**. Used for:

    - Creating and managing Property Managers
    - Listing Property Managers
    - Configuring push notification endpoints
    - Retrieving channel configuration

    ```bash
    # PMS-level login
    curl -G "https://apidemo.mybookingpal.com/authc/login" \
      --data-urlencode "username=pms@yourcompany.com" \
      --data-urlencode "password=pms-password"
    ```
  </Tab>
  <Tab title="PM-Level Token" icon="user">
    Obtained using **PM credentials**. Used for:

    - Managing products (properties)
    - Setting rates, availability, and LOS pricing
    - Managing images, fees, and taxes
    - Handling reservations and messaging

    ```bash
    # PM-level login
    curl -G "https://apidemo.mybookingpal.com/authc/login" \
      --data-urlencode "username=pm-email@example.com" \
      --data-urlencode "password=pm-password"
    ```
  </Tab>
</Tabs>

## Webhook Authentication (Incoming)

Push notification endpoints (webhooks) sent **from** BookingPal to **your** server use a different verification mechanism:

<Columns cols={2}>
  <Card title="Reservation Push" icon="shield" horizontal="false">
    Verified via the `x-api-key` header included in the request. Confirm the notification is from BookingPal by validating this key.
  </Card>
  <Card title="Async Push" icon="zap" horizontal="false">
    No authentication. These are informational notifications sent to your `asyncPush` endpoint.
  </Card>
</Columns>

## Error Handling

| Status Code | Description |
|---|---|
| `422` | Username or password is empty |
| `401` | Invalid credentials, expired token, or missing `x-api-key` header |
| `403` | Insufficient permissions for the requested operation |

<Expandable title="Why am I getting a 401?" default-open="false">

Verify that:

- The `x-api-key` header is included in every request.
- The token is valid and has not expired (120-minute lifetime).
- The token is passed as the `jwt` **query parameter**, not in headers.
- You are using the correct credential type (PMS vs PM) for the endpoint.
- The URL is correctly encoded when passing the token.

</Expandable>

<Expandable title="How do I handle token expiration gracefully?" default-open="false">

Recommended approach:

1. Store the token and the timestamp when it was obtained.
2. Before each API call, check if the token is older than ~110 minutes.
3. If so, re-authenticate to get a fresh token.
4. If an API call returns 401, automatically re-authenticate and retry the request once.

</Expandable>
